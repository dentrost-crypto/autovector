# AutoVector AI Context

## 1. Цель проекта

AutoVector — проект по продаже автомобилей из Китая, Кореи и Японии под заказ в Россию.

Главная задача проекта — создать цифровую инфраструктуру, которая помогает:

* собирать автомобили из внешних источников;
* показывать их на сайте-витрине;
* автоматически создавать продающий контент;
* публиковать автомобили в Telegram;
* получать заявки от клиентов;
* передавать клиента менеджеру для расчёта и сделки.

AutoVector не должен выглядеть как случайная копия чужого каталога. Это должна быть отобранная витрина автомобилей для российского покупателя.

## 4. Текущая архитектура

### Сайт

Сайт работает как витрина автомобилей.

Основные данные автомобилей лежат в:

app/data/cars.json

Состояние опубликованных автомобилей хранится в:

app/data/posted-cars.json

Фотографии автомобилей находятся в:

public/uploads/

и частично:

public/uploads/cars/{id}/

MP4-видео создаются в:

public/generated/videos/

## 5. Telegram-автопостинг

Автопостинг работает через GitHub Actions.

Workflow:

.github/workflows/post-cars-to-telegram.yml

Скрипт публикации:

scripts/post-to-telegram.js

Логика:

1. Скрипт читает cars.json.
2. Скрипт читает posted-cars.json.
3. Выбирает следующий неопубликованный автомобиль.
4. Проверяет наличие MP4.
5. Если MP4 нет — вызывает генератор видео.
6. Публикует видео и текст в Telegram.
7. Добавляет ID автомобиля в posted-cars.json.
8. GitHub Actions коммитит posted-cars.json обратно в репозиторий.

Важно:
posted-cars.json нужен для защиты от дублей. Его нельзя случайно очищать.

---

## 6. Видео

Генератор видео:

scripts/build-car-video.js

Видео создаётся автоматически перед публикацией, если для автомобиля ещё нет MP4.

Формат:

* вертикальное видео 1080x1920;
* длительность около 13 секунд;
* используется FFmpeg;
* верхний брендинг AutoVector;
* первый крупный экран — название автомобиля;
* далее эмоциональные триггеры;
* финальный экран — цена и бренд AutoVector.

---

## 7. Правила видео AutoVector

Видео должно продавать эмоцию, а не технические характеристики.

Технические характеристики уже есть в текстовом Telegram-посте и на сайте.

В видео нельзя перегружать экран цифрами и длинными фразами.

### Основные правила

1. Первый крупный текст — название марки / модели автомобиля.
2. В верхней части ролика постоянно показывать AutoVector.
3. Использовать максимум 3 эмоциональных триггера на ролик.
4. На одном экране максимум 2–4 слова.
5. Триггеры должны ротироваться, чтобы ролики не выглядели одинаковыми.
6. Финальный экран должен содержать цену и AutoVector.

## 10. Правило ротации триггеров

Если в предыдущем ролике использовались:

* премиальный стиль
* уверенная динамика
* комфорт каждый день

то в следующем похожем ролике нельзя использовать тот же набор.

Нужно выбрать синонимы из тех же категорий:

* статусный выбор
* характер в движении
* уют в поездке

Цель:
публикации не должны выглядеть однообразно.

## 11. Тексты Telegram-постов

Telegram-посты должны быть понятными для обычного покупателя, а не для автоэксперта.

Целевая аудитория:

* женщины 35–55 лет;
* семьи;
* люди, которым нужен красивый, надёжный, понятный автомобиль;
* покупатели среднего и выше среднего класса;
* люди, которые не хотят разбираться в сложных технических деталях.

В начале поста должны быть:

* название авто;
* цена под ключ в РФ;
* год;
* пробег.

Цена должна быть заметной.

## 13. Стратегия парсинга

Текущий следующий этап — многоссылочный пакетный парсинг Korex.

Идея:

Одна ссылка = одна модель или один фильтр на Korex.

Пример:

* Toyota Corolla
* Toyota Camry
* Toyota RAV4
* Toyota Highlander
* Toyota Avalon

Для каждой ссылки:

1. открыть страницу выдачи;
2. собрать карточки автомобилей;
3. взять максимум 5 автомобилей;
4. перейти в каждую карточку;
5. собрать данные;
6. скачать фото;
7. добавить автомобили в общий каталог.

Один запуск может обрабатывать несколько ссылок подряд.

Пример:

5 ссылок × 5 автомобилей = до 25 автомобилей за один запуск.

---

## 14. Важное правило парсинга

Парсер не должен без необходимости удалять текущий каталог.

По умолчанию нужно использовать безопасный режим:

APPEND_MODE = true

Новые автомобили добавляются к существующим.

Полная замена каталога допускается только отдельным явным режимом.

## 15. Фото

Для MVP фотографии можно хранить локально в проекте.

Основные пути:

public/uploads/

public/uploads/cars/{id}/

Рекомендация:

* не хранить слишком много фото на автомобиль;
* достаточно 4–6 фото;
* использовать WebP;
* сжимать изображения;
* не коммитить лишние временные файлы, если они не нужны сайту или автопостингу.

## 16. Текущая приоритетная задача

Реализовать многоссылочный пакетный парсинг Korex.

Нужно сделать так, чтобы парсер мог принимать список источников:

[
{
"name": "Toyota Corolla",
"url": "...",
"limit": 5
},
{
"name": "Toyota Camry",
"url": "...",
"limit": 5
}
]

И обрабатывать их по очереди.

Главная цель:
быстро расширить каталог AutoVector до 50–100 автомобилей разных брендов и моделей, не ломая текущую цепочку:

Парсер
→ Excel
→ cars.json
→ сайт
→ MP4
→ Telegram

---

## 17. Что нельзя ломать

При любых изменениях нельзя ломать:

* текущий сайт;
* структуру cars.json;
* Telegram autoposting;
* posted-cars.json;
* генерацию MP4;
* GitHub Actions;
* существующие карточки автомобилей.

Любые изменения должны быть минимальными и проверяемыми.

---

## 18. Рабочий стиль

Перед изменением сложной логики сначала делать диагностику и отчёт.

Затем вносить минимальное изменение.

После каждого изменения показывать:

* какие файлы изменены;
* что проверено;
* что не трогалось;
* какие риски остались.
